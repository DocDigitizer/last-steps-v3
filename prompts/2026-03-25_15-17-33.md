Actually we already have a definition of what a schema is and we already have some of them. The definition of a schema for us is something like this:  a schema is a versioned JSON schema that defines the expected data structure for extracting information from a specific type of document, optionally scoped to a country.

  Key characteristics:

  - Purpose: It tells a document extraction pipeline what fields to extract from a document (e.g., an Invoice schema for Portugal might include a nif field for the Portuguese tax ID).
  - Scoped by doc type and country: Every schema is linked to a document type (Invoice, CV, Contract…) and optionally to a country. A country-specific schema captures locale-specific fields; a generic one (no
  country) serves as a fallback.
  - Versioned: Schemas are versioned under a stable publicId (sch_xxx), with each version getting its own publicVersionId (schv_xxx). Updates to an active schema create a new version rather than mutating in
  place.
  - Lifecycle: draft → active → deprecated. Only active schemas are returned by the selection endpoint.
  - Visibility: Can be public (available to everyone, requires a country), community (logged-in users), or private (owner org only).
  - Selection: The find-best endpoint picks the best schema for a given doc type + country by prioritizing the org's own schemas first, then public exact-country matches, then public generic (no country) as
  fallback.

  In short: a schema is the extraction template — it describes what structured data should come out of a particular kind of document.
So having this in mind what thing should we be looking at that this structure does not already implement and we are not looking and stuff like that?
